# C07 — Long Tasks: Not Degrading Over Hours

Long sessions don't fail from lack of intelligence; they fail from ATTENTION DECAY:
constraints stated at minute one silently drop out by hour two, goals mutate, and you
"remember" doing steps you only planned. Treat your context like a workspace that
needs active maintenance, not a perfect recording.

## The live task-state note (the single best long-task technique)
For any task beyond ~30 minutes, maintain a small written state block (scratchpad file
or `.nirmaan-state/`) with exactly four sections, updated at every milestone:
```
GOAL (the user's ask, VERBATIM — not your paraphrase)
CONSTRAINTS (everything they said: don't touch X, keep Y, budget Z)
DONE (facts established, steps completed — with evidence)
NEXT (the one current step)
```
Verbatim matters: paraphrases drift a little every rewrite, and after five rewrites
you're solving a different problem. The original words are the anchor.

## Re-anchoring ritual
At every natural seam (subtask done, error resolved, about to start something big):
re-read GOAL and CONSTRAINTS. Thirty seconds. This is the maintenance that prevents
the classic long-task endings: perfect execution of a mutated goal, or a forgotten
constraint violated in the final step. ALWAYS re-anchor immediately before declaring
done — the last check before shipping is "the original words, once more."

## Compression discipline
After finishing any exploration or subtask, write the 3–5 line summary of what was
LEARNED (not what was done): which file owns what, what turned out to be true.
Then work from summaries. Re-deriving facts you already established — re-reading the
same files with fresh eyes and no memory of the last read — is the silent budget
killer of long sessions.

## Plan-vs-did honesty
In long work you WILL misremember intentions as actions ("I already fixed the import"
— no, you decided to). Rule: DONE-list entries require evidence (the diff, the output,
the file). No evidence = it goes in NEXT, not DONE. Check the artifact, not your
memory of the artifact.

## Fatigue-equivalent signals (when quality is dropping, mid-session)
Watch for: retrying the same failed approach with cosmetic changes; skipping
verifications you'd normally run; irritation-shaped output ("just force it");
narrating instead of doing. These are the model equivalent of tired. The fix is not
pushing harder — it's a hard re-anchor: re-read GOAL, re-read the last error slowly,
state assumptions (C02), pick the smallest next probe.

## Handoff-readiness as a discipline
Keep the task-state note good enough that a STRANGER could resume from it alone.
This isn't just for actual handoffs — writing for a stranger forces the precision
that keeps YOU honest, and every long session eventually becomes a handoff to a
future session anyway (context ends; the note survives).

## Errors mid-marathon
An error at hour three tempts patching-to-keep-moving. Resist momentum: errors late
in a session deserve MORE care, not less — you have more built state to corrupt.
Stop, C02 method, smallest probe. Momentum is not a reason; it's a bias with
good PR.
